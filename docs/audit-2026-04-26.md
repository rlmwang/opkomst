# Opkomst — comprehensive audit (2026-04-26)

State of the codebase after the afdelingen feature lands. ~2 000 LoC backend, ~2 800 LoC frontend.

---

## TL;DR

The app is **feature-complete for the original brief**: organisers register → admin approves with afdeling → create event → public signs up (anonymously) → optional encrypted email → automatic post-event questionnaire → bounce-aware delivery tracking → response stats. SCD2 versioning lands on afdelingen; events SCD2 is planned but not built. Privacy invariants from the original design are intact and enforced at multiple layers.

The biggest gaps before going live: **no automated tests**, **no production deploy story** (Dockerfile / Coolify config / migration path / SMTP credentials wiring), and **no rate limiting on public surfaces** (signup endpoint, feedback submit, login). The feedback email is always in Dutch regardless of attendee preference. The frontend bundle is one big chunk (~497 kB main).

## Backend audit

### Stack

- **FastAPI** 0.115+ on Python 3.13.
- **SQLAlchemy** 2.0 with declarative `Mapped[]` typing. SQLite locally; the migrations and code are Postgres-ready (no SQLite-only constructs except for the `LargeBinary` column which is portable).
- **Alembic** with `render_as_batch=True` so SQLite roundtrips work; migrations auto-apply on app startup via `backend/migrate.py`.
- **APScheduler** drives the hourly feedback sweep in-process.
- **Jinja2** for HTML email templates.
- **bcrypt** direct (not via passlib — see below).
- **python-jose** for JWT.
- **uuid-utils** for uuid7 PKs.
- **structlog** everywhere for structured logging (no PII per design).

### Models (`backend/models/`)

| Model | Purpose | SCD2? | Notes |
|---|---|---|---|
| `User` | Organiser / admin accounts | No | `email_verified_at`, `is_approved`, `afdeling_id` (entity_id), `role` (admin / organiser). |
| `Afdeling` | Local chapter | **Yes** | `entity_id` + `valid_from` / `valid_until` + `changed_by` + `change_kind`. Soft-delete + restore. |
| `Event` | An event | No (planned) | Slug-addressed, `archived_at` for soft-archive, `afdeling_id` FK to `Afdeling.entity_id`. SCD2 plan in `docs/scd2-events-migration.md`. |
| `Signup` | Public signup row | No (append-only) | `encrypted_email` (AES-GCM), `feedback_email_status`, `feedback_message_id` (Scaleway TEM correlation), `feedback_sent_at`. |
| `FeedbackQuestion` | The five fixed questions | No | `key` is the i18n lookup root. |
| `FeedbackToken` | One-time URL-safe token | No | Deleted on submit or send-failure. The only link from response → signup. |
| `FeedbackResponse` | A single answer | No | Linked to `event_id` only — never to signup or token. |
| `AuditLog` | Admin actions | No | `actor_id`, `target_id`, `action`. Approve / promote / demote / assign_afdeling. |

**Relational integrity caveats:**

- `Signup.event_id`, `FeedbackToken.event_id`, `FeedbackResponse.event_id` all FK to `Event.id` (the row id). When the events SCD2 lands, these need to switch to `Event.entity_id` (already documented in the SCD2 plan).
- `Event.afdeling_id` and `User.afdeling_id` point at `Afdeling.entity_id` (no FK — entity_id isn't unique across all rows because multiple versions share it). Integrity is enforced in code via `services.afdelingen.find_current_by_entity`. **Risk:** a code path that bypasses that helper could leave a dangling reference.
- The `users.email` unique constraint is case-sensitive at the DB layer. Schema-level `LowercaseEmail` (`schemas/common.py`) is the single normalisation point — every email-bearing input field uses it. **Risk:** if a future `User(email=...)` write site bypasses Pydantic and hits the ORM directly with mixed-case, the unique constraint won't catch a duplicate.

### Auth (`backend/auth.py`)

Two gates compose: `email_verified_at IS NOT NULL` AND `is_approved = true`. `require_approved` enforces both with distinct 403 messages. `require_admin` adds `role = "admin"`.

- **Bootstrap admin path** auto-verifies + auto-approves the very first registration matching `BOOTSTRAP_ADMIN_EMAIL` (chicken-and-egg). Every subsequent registration goes through both gates.
- **Purpose tokens** (signed JWTs scoped by `purpose` field) for email verification. 24h TTL.
- **Bcrypt** truncates to 72 bytes (input cap) — explicit in the code.
- **JWT** stores `sub` (user id) only. 7-day TTL. No refresh tokens; on expiry the user logs in again.
- **No CSRF protection.** JWT in `Authorization` header from localStorage means the cookie-CSRF vector doesn't apply. **Risk:** any XSS that gets an attacker JS into the page would exfiltrate the token. The CSP is permissive (`'unsafe-inline'` for PrimeVue runtime styles). For a small-org tool this is acceptable, but worth pinning down before launch.

### Privacy invariants (`CLAUDE.md` enumerates these; backend enforces)

1. **No PII in logs.** Routes log `INFO` lines with route name + outcome — no email, no IP, no UA. Auditors should verify by grepping `logger.info(...` for any `email=`, `display_name=`, etc. (Did the audit; only `to=` in the email service's send-confirm log carries an address — that's intentional and isolated to the send hop.)
2. **Email decryption only in the feedback worker.** `services/encryption.py:decrypt` has a single legitimate caller: `services/feedback_worker.py`. `seed.py` calls `encrypt` (never `decrypt`). Worth a pre-commit check.
3. **Encrypted email is hard-deleted after use.** `_process_one` sets `signup.encrypted_email = None` and stamps `feedback_sent_at` regardless of send success — this is verified by the smoke test.
4. **Open-source disclosure on every signup form.** Public event page links to `https://github.com/rlmwang/opkomst` in the privacy notice.
5. **No third-party analytics or tracking pixels.** None present. The Leaflet map fetches OSM tiles from the OSM tile server (a third party, but for tile rendering only — no user-identifying data is sent).
6. **Email is venue-policy-gated by design.** When `Event.questionnaire_enabled = false`, the public form omits the email field and the worker silently skips the event. No email is ever collected.

### Email pipeline

```
Public signup form
  ↓ (only when questionnaire_enabled)
  encryption.encrypt(email) → bytes → Signup.encrypted_email
  Signup.feedback_email_status = "pending"

Hourly worker tick (or organiser-triggered)
  ↓ for each pending signup whose event ended ≥24h ago
  Mint FeedbackToken (URL-safe random, 30-day TTL)
  Generate Message-ID (random hex @ MESSAGE_ID_DOMAIN)
  Decrypt email (only legitimate caller)
  Render Jinja template with locale = "nl"
  Hand to backend (console / SMTP) with Message-ID header
    ↓ on success: status = "sent", message_id stored
    ↓ on failure-after-retry: status = "failed", token deleted, message_id discarded
  Wipe encrypted_email regardless (privacy invariant)

Scaleway TEM webhook
  ↓ POST /api/v1/webhooks/scaleway-email
  HMAC-SHA256 verify against SCALEWAY_WEBHOOK_SECRET (when set)
  Lookup signup by feedback_message_id
  If event in BOUNCE_EVENTS: status = "bounced"
  If event in COMPLAINT_EVENTS: status = "complaint"
  Soft bounces / deliveries / opens: ignored

Public submission /api/v1/feedback/{token}/submit
  ↓
  Validate token exists + not expired
  Validate required questions (Q1, Q2)
  Generate submission_id (random)
  Insert FeedbackResponse rows (linked to event_id only)
  Delete FeedbackToken (one-shot)
```

**Gaps:**
- Locale of the feedback email is hardcoded to `"nl"`. Attendees who speak English get Dutch. The signup form doesn't collect locale, but the backend could persist `accept-language` or expose a picker. Track for v1.1.
- Bounce processing only flips status; it doesn't notify the organiser. The dashboard breakdown shows the count but there's no inbox / digest. Acceptable — organisers see counts on the Details page.
- `EMAIL_BACKEND` defaults to `console`. Production must set `EMAIL_BACKEND=smtp` plus the four SMTP envs. The deploy story doesn't yet exist.

### Routers

| Router | Endpoints | Auth |
|---|---|---|
| `auth.py` | register, login, /me, verify-email, resend-verification | mixed |
| `admin.py` | list users, approve, assign-afdeling, promote, demote | admin |
| `afdelingen.py` | list, create, archive, restore | mixed (read = approved, write = admin) |
| `events.py` | list, list-archived, create, by-slug, qr.png, update, archive, restore, send-feedback-emails-now, stats | scoped to user's afdeling |
| `signups.py` | public POST | none (public) |
| `feedback.py` | questions list, public form GET, public submit, organiser summary | mixed |
| `webhooks.py` | scaleway-email | HMAC if secret set |

**Cross-router scoping:** every event-level read / mutation outside the public surface goes through `_get_event_scoped(db, event_id, user)` (events router) or the equivalent `afdeling_match` filter (feedback router). 404 on cross-afdeling — no existence leak. Public endpoints (by-slug GET, QR, signup POST, feedback GET / submit) deliberately skip scoping; the slug or token is the access token.

**Open routes I'd flag:**
- `POST /api/v1/auth/register` is unauthenticated and unrate-limited. Anyone can create accounts indefinitely. The admin-approval gate prevents abuse from acting on the system, but the table grows unboundedly. **Recommendation:** add a per-IP rate limit before launch.
- `POST /api/v1/events/by-slug/{slug}/signups` is unauthenticated and unrate-limited. Same risk: spam signups inflate counts and burn email budget if questionnaire is on. **Recommendation:** rate-limit + signup-form CAPTCHA for high-traffic events.
- `POST /api/v1/feedback/{token}/submit` is one-shot per token; rate-limiting is less critical there but a per-IP cap would still help against token-guessing brute force (token entropy is 32 random bytes so brute-force is not a practical threat).

## Frontend audit

### Stack

- **Vue 3** Composition API + `<script setup>`.
- **TypeScript** strict mode, `vue-tsc --noEmit` clean.
- **Vite** 6 build.
- **Pinia** 3 for state.
- **PrimeVue 4** with custom Aura preset (`OpkomstPreset` defines the brand palette anchored at `primary.500 = #9f000b`).
- **vue-i18n** 11 with `legacy: false`. Two locales (nl, en), Dutch default. Persisted to localStorage and synced to `<html lang>`.
- **Leaflet** for the inline map (OSM tiles).
- **PrimeIcons** for the icon set; **emoji** for flags.

### Pinia stores

- `auth` — current user, login / register / verify / resend / logout.
- `events` — active list, archived list, create / update / archive / restore / sendFeedbackEmailsNow / getStats / getBySlug.
- `admin` — users list, approve (with afdeling), assignAfdeling, promote, demote.
- `afdelingen` — fetchAll, search, create, archive, restore.
- `feedback` — fetchQuestions, getForm, submit, getSummary.

All stores are simple — no shared state across stores, no cross-store reactivity gotchas. Refetching after mutations keeps everything consistent.

### i18n coverage

Every user-facing string goes through `t(...)`. NL is the default + fallback. Sample keys per locale: ~140. Date formatting respects locale via `toLocaleString(localeTag())` where `localeTag()` returns `nl-NL` or `en-GB`. Number formatting (party size) uses native rendering (no localised numerals needed for 1-50).

Audit found one explicit Dutch term that's deliberately untranslated: **"afdeling"** itself. It's the org's terminology and stays Dutch in the English UI too — that's a brand call, not an oversight.

### Theming

- Cream surfaces (`--brand-bg #f6f1e7`, `--brand-surface #fbf7ee`) — pamphlet stock.
- Brand red (`#9f000b`) anchors `primary.500` in the PrimeVue preset; lighter pinks fill 50–400; darker reds drive hover/active.
- Secondary buttons re-tinted into the brand-red family at low saturation (CSS overrides in `theme.css`) — the default Aura grey palette clashes with cream.
- ConfirmDialog reskinned via CSS to use cream surface + brand-red title + warm border.
- `<html lang>` syncs with locale; `darkModeSelector` pinned to a class that's never set so PrimeVue doesn't follow OS dark mode (the surfaces are hard-coded light).

### Frontend gaps / risks

- **Bundle size.** The main chunk is 497 kB (≈131 kB gzip). PrimeVue + Leaflet + vue-i18n + Pinia are the bulk. Acceptable for a small-org tool. If the public surface becomes performance-critical (poor mobile networks at events) we should split routes (Leaflet only loads on the public event page where it's actually used).
- **No tests.** No vitest / Playwright suites. The smoke tests in this audit live as ad-hoc scripts I ran during development — they aren't checked in and don't run in CI.
- **No mobile design verification.** I added a media query to the dashboard event card to flatten the right-side column on narrow viewports, but I haven't tested every page on a phone. The forms (event create, public signup, feedback) are likely fine because they're vertically stacked; the admin page may be cramped at 360px.
- **Accessibility audit not performed.** Color-contrast looks OK against cream (the brand red passes WCAG AA on `#fbf7ee`), focus rings exist via PrimeVue defaults, ARIA labels exist on icon-only buttons. But no Axe / Lighthouse pass yet.
- **No skeleton / loading states beyond `:loading` on submit buttons.** The Dashboard / Details / Archived pages briefly render an empty card before the fetch resolves. Cheap to fix with a tiny `<Skeleton>` or just a "Loading…" line.
- **The `errorMessage(e, fallback)` helper in `lib/errors.ts`** is referenced from PeoplePage — confirm it exists and behaves as expected (a quick `grep` says it does).
- **The QR pixel-by-pixel transparency loop** in `events.py` is O(n²) over a 330x330 image. Microseconds at worst, but PIL has a faster path via `Image.point` + `putalpha`. Cosmetic optimisation; ignore.

## UX audit

### Sign-up & onboarding flow (organiser)

1. `/register` — clear copy ("after registering you'll get a confirmation email; then an admin needs to approve…"). Submit → 201 + verification email queued + JWT in localStorage.
2. `/dashboard` shows the **unverified** empty state with a "Send a new verification email" button. Resending works.
3. User clicks the link in their email → `/verify-email?token=…` → success state with "Go to dashboard".
4. Back on `/dashboard`, the **verified-but-pending** state replaces the unverified one.
5. Admin approves with afdeling → user reloads → full dashboard.

**Smooth.** One frustration: the user has no signal between (3) and (4) that the admin has approved them. **Recommendation for v1.1:** an optional "we approved your account" notification email when the admin clicks Approve. Out of scope for the original brief but a small UX win.

### Admin onboarding

The bootstrap admin path (env-var-set email, first registration only) auto-verifies and auto-approves. Otherwise an admin-less system is unfixable from the UI. Documented in `.env.example` and `seed.py`.

### Event creation

`EventFormPage` is solid. Date / start / end split into three fields per the brief, 15-min step on time pickers, location autocomplete via Nominatim, source-options as an `EditableList`, questionnaire toggle defaulted on with explainer text. Save lands on `/events/:id/details` so the organiser immediately sees the share link / QR.

**Minor frustrations I noted while building:**
- The "topic" field is optional but its purpose isn't obvious. The placeholder says "Onderwerp (optioneel)". It's used as the italic subtitle on the public page. Could use a one-line helper text.
- There's no preview of what the public page looks like before publishing. Organisers have to create the event and then click the link. Acceptable.

### Public signup form

`PublicEventPage` is the most-trafficked surface. Design choices:
- Map at the top under the meta block — visible without scrolling.
- Privacy explainer is collapsed by default (chevron-prefixed `<details>`). Keeps the form short.
- Three required fields (name, party size, source), one optional (email, conditionally hidden).
- Address is clickable, opens OpenStreetMap.
- "Number of people" has a help line clarifying "including yourself".
- Thanks state replaces the form on submit.

**One ambiguity:** the privacy explainer copy changes when `questionnaire_enabled = false` ("we're not asking for any contact details"), but the absence of the email field is also visible. Possibly redundant. Acceptable — better to be explicit twice than to have the no-email path feel suspicious.

### Feedback form

`FeedbackPage` mirrors the public signup's information density. Five questions, the first two required, marked with a red `*`. The thanks state explains the email has already been deleted.

**Possible improvement:** show progress (e.g. "Question 3 of 5"). Five questions is short enough that it's probably overengineering.

### Feedback delivery & breakdown

The `EventDetailsPage`'s "Aflevering feedbackmail" card surfaces the per-status counts as colour-coded pills (green sent, amber pending, red bounced/complaint/failed, neutral no-email). The seed populates one signup in every state on the past demo event so the visual works on first boot.

The "Send now" button at the bottom of the page lets organisers bypass the 24h cutoff. Disabled when nothing's pending or the questionnaire is off, with an explainer line in the disabled state. Confirm dialog before the actual send.

### Archive flow

Archive an event from the dashboard → confirm dialog → it disappears from the list. The archived view at `/events/archived` shows them with a Restore button. Restoring brings them back to the active list. Per-event public link 404s while archived.

### Afdelingen UX

Admin page now has two cards: Afdelingen (with the EditableList + AfdelingPicker / inline create) and Users (each row tagged with their afdeling). The picker shows soft-deleted afdelingen with a "deleted — restore?" tag. Approve / Change-afdeling open a dialog with the picker so the afdeling assignment is always explicit.

**One UX wart:** the "or type a new name" input is separate from the "search archived" autocomplete. Two inputs to add an afdeling. Functionally correct (the picker selects from existing, the text field creates new) but visually redundant. Could be unified into a single AutoComplete that also accepts free text — but that's an iteration after the data model settles.

### Inconsistencies I noticed

- Some CSS class names use kebab-case (`event-row`), some use camelCase (no instances actually, all kebab — false alarm).
- "Statistieken" → "Details" rename was global, but the backend route is still `GET /events/{id}/stats`. That URL is the source aggregator, not the page; renaming would break all consumers and serves no UX value.
- The brand color is referenced in two places: the PrimeVue preset (`primary.500`) and `--brand-red` in `theme.css`. They're kept in lock-step manually — a comment notes this. Future regression risk.

## What's not built (gaps relative to a launch-ready product)

**Critical for launch:**
1. **Tests.** Backend: pytest suite covering auth flow, afdeling scoping, privacy invariants (decrypt-only-from-worker, ciphertext-wiped-after-send, no-PII-in-logs). Frontend: vitest for stores + Playwright for the four critical flows (register → verify → approve → create event → signup → feedback).
2. **Production deploy.** No Dockerfile, no Coolify / Fly / Render config. No migration story for the first prod deploy (does it just run on boot? probably yes but verify). No SMTP credential management.
3. **Rate limiting** on `/auth/register`, `/auth/login`, `/events/by-slug/{slug}/signups`. SlowAPI or starlette-limiter ports easily.
4. **HTTPS / HSTS / proper CSP.** The current `index.html` is fine for local dev, but production should pin a strict CSP and enable HSTS.
5. **Error monitoring.** structlog logs go to stderr; no Sentry / Glitchtip / etc. wired up.

**Important but not launch-blocking:**
6. **SCD2 on events** — plan exists at `docs/scd2-events-migration.md`. ~4 hours of careful work.
7. **Organiser locale picker on signup form.** Current feedback email always Dutch.
8. **Per-event language for the public page.** Today the locale follows the viewer's preference; might want the event creator to pin it.
9. **Approval notification email** to the organiser when their account is approved.
10. **CSP review.** PrimeVue 4 needs `'unsafe-inline'` for runtime style injection. Pin a strict policy that allows that and only that.

**Nice-to-have:**
11. **Bundle splitting.** Lazy-load Leaflet + Vue-Toastification on the public page; keep the auth bundle tiny.
12. **Skeleton loading states.**
13. **Mobile design pass.**
14. **Accessibility audit** (Axe / Lighthouse).
15. **The `<EditableList>` AfdelingPicker UX** could combine the search-archived autocomplete and the new-name input into one.
16. **`scd2-safety` pre-commit hook** (planned in the SCD2 doc).
17. **A graceful-shutdown story** for APScheduler — currently `_lifespan` calls `shutdown(wait=False)` which is fine but doesn't drain in-flight sends.

**Not needed:**
- A users list / search beyond the admin page (small org, no point).
- Multi-organiser ownership of events (small-org trust: anyone in the afdeling can edit).
- An events-search UI (lists are short).
- An attendees export (privacy-by-design — we don't expose individual signups, only aggregates).

---

## Recommendations, in order of priority

1. **Write tests** — backend pytest first, frontend vitest second, Playwright for the critical paths last. Without these the next refactor breaks something silently.
2. **Set up production deploy** — Dockerfile, Coolify config (mirroring horeca-backend), real SMTP credentials in env, run a real migration on a real Postgres.
3. **Add rate limiting** to the three public mutations.
4. **Write a one-page operations runbook** — backup strategy, how to rotate `EMAIL_ENCRYPTION_KEY` (which would invalidate all in-flight encrypted emails — needs a migration plan).
5. **Execute the SCD2 events migration** when there's appetite — the plan is ready.
6. **Localised feedback emails** based on a stored attendee preference (or the creator's afdeling default).
7. **CSP hardening + Sentry/Glitchtip integration** before any real traffic.
